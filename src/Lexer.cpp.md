# Lexer.cpp - The Quantum Language Compiler's Lexical Analyzer

## Overview

`Lexer.cpp` is a fundamental part of the Quantum Language compiler, tasked with performing the lexical analysis phase. This phase converts the input source code into a sequence of tokens, which serves as the basis for further parsing by the compiler. The lexer handles various lexical elements such as keywords, operators, identifiers, literals, and comments.

### Role in Compiler Pipeline

The lexer operates at the beginning of the compilation process. It reads the source code character by character, identifies meaningful units (tokens), and categorizes them according to their syntactic roles. These tokens are then passed to the parser for semantic analysis and further processing.

### Key Design Decisions and Why

1. **Tokenization Rules**: The lexer uses predefined rules to identify different types of tokens. For example, it recognizes keywords like `let`, `const`, and `fn`, and treats them as distinct token types (`TokenType::LET`, `TokenType::CONST`, etc.).

2. **Error Handling**: To ensure robustness, the lexer includes error handling mechanisms. If an unexpected character or syntax is encountered, it throws exceptions that can be caught and handled by higher-level components of the compiler.

3. **State Management**: The lexer maintains state information such as the current position in the source code, line number, and column number. This allows it to accurately report errors and provide context during debugging.

4. **Efficiency**: The implementation aims to balance efficiency with simplicity. By using straightforward algorithms and data structures, the lexer ensures quick tokenization without sacrificing clarity or maintainability.

## Major Classes/Functions Overview

### Lexer Class

- **Constructor**: Initializes the lexer with the source code string.
- **current()**: Returns the current character being processed.
- **peek(int offset)**: Returns the character at a specified offset ahead in the source code.
- **advance()**: Advances the lexer to the next character and updates the line and column numbers accordingly.
- **skipWhitespace()**: Skips over any whitespace characters until a non-whitespace character is found.
- **skipComment()**: Skips over single-line comments starting with `//`.
- **skipBlockComment()**: Skips over multi-line block comments enclosed within `/* */`.

### Token Class

- Represents individual tokens generated by the lexer.
- Contains attributes such as the token type, value, and location in the source code.

### TokenType Enum

- Defines all possible token types recognized by the lexer.
- Includes categories like keywords, operators, identifiers, literals, and special symbols.

## Tradeoffs

1. **Flexibility vs. Simplicity**: While the lexer supports a wide range of token types, its simplicity has been prioritized to enhance maintainability and readability.

2. **Performance vs. Complexity**: The lexer is designed to be efficient, but some complex edge cases might require additional performance optimizations or complexity in the codebase.

3. **Error Reporting vs. User Experience**: Accurate and detailed error reporting helps in diagnosing issues quickly, but it might sometimes lead to verbose error messages that could confuse users.

By addressing these tradeoffs, the Quantum Language compiler aims to provide a reliable and user-friendly development environment for quantum programming.
